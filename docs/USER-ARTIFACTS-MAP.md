# User-Defined Artifacts Map

Audit of every file in career-ops that contains or should contain user-specific data.
Starting point for iteration 1: making the system pluggable for multiple users.

---

## The Core Problem

User data is scattered across 8+ locations at different directory levels with no single entry point.
To onboard Cuti today, you'd have to hunt and replace across all of these manually.
The goal of iteration 1 is to identify all of it, consolidate what can be consolidated, and define
a clean boundary so Cuti (or a future CLI flag) just points to their own directory.

## Issues Found

### Critical

| # | Issue | Impact |
|---|-------|--------|
| 1 | `modes/_profile.md` still has Santiago's AI archetypes | Every evaluation scores against the wrong archetypes. the user's DoD profile is NOT driving scoring. |
| 2 | `voice-dna.md` is Santiago's personal voice file | Cover letters and outreach will imitate Santiago's voice, not the user's. |
| 3 | `article-digest.md` doesn't exist | Modes reference it constantly for proof points. Missing file = generic outputs with no metrics. |
| 4 | CV variants not in `DATA_CONTRACT.md` | Undeclared user files — invisible to update protection. |

### Structural (Pluggability)

| # | Issue | Impact |
|---|-------|--------|
| 5 | User data spread across root, `config/`, `modes/`, `data/`, `interview-prep/`, `writing-samples/`, `jds/` | Impossible to "switch users" or onboard someone new without touching 7+ directories. |
| 6 | `data/Company Watchlist.md` not in DATA_CONTRACT | Undeclared. Update protection doesn't know it exists. |
| 7 | No template for `voice-dna.md` | New users have no starting point — they get Santiago's voice or nothing. |
| 8 | No template for `article-digest.md` | New users don't know the format until they find `examples/article-digest-example.md`. |
| 9 | `auth/linkedin_cookies.json` undocumented | LinkedIn scanning referenced in portals but no setup docs for the auth file. |
| 10 | Multi-CV pattern (cv-pm-dod.md, etc.) is undocumented | Custom pattern we invented — no naming convention or system awareness. |

## Current File Locations vs. Ideal Pluggable Layout

```
CURRENT (scattered)                    IDEAL (iteration 1 goal)
─────────────────────────────────────  ──────────────────────────────────────
cv.md                              →   users/{name}/cv.md
cv-pm-dod.md                       →   users/{name}/cv-pm-dod.md
cv-sm-dod.md                       →   users/{name}/cv-sm-dod.md
cv-consultant.md                   →   users/{name}/cv-consultant.md
voice-dna.md                       →   users/{name}/voice-dna.md
article-digest.md                  →   users/{name}/article-digest.md
portals.yml                        →   users/{name}/portals.yml
config/profile.yml                 →   users/{name}/profile.yml
modes/_profile.md                  →   users/{name}/_profile.md
data/applications.md               →   users/{name}/data/applications.md
data/pipeline.md                   →   users/{name}/data/pipeline.md
data/scan-history.tsv              →   users/{name}/data/scan-history.tsv
data/follow-ups.md                 →   users/{name}/data/follow-ups.md
data/Company Watchlist.md          →   users/{name}/data/company-watchlist.md
interview-prep/                    →   users/{name}/interview-prep/
writing-samples/                   →   users/{name}/writing-samples/
reports/                           →   users/{name}/reports/
output/                            →   users/{name}/output/
jds/                               →   users/{name}/jds/
auth/linkedin_cookies.json         →   users/{name}/auth/linkedin_cookies.json
.env                               →   users/{name}/.env  (or root .env)
```

Second user would be: `users/{other}/` — completely isolated, no overlap.

## Immediate Gaps to Fill (Before Next Scan)

Priority order for the user to be fully operational:

1. **Populate `modes/_profile.md`** with the user's actual DoD archetypes — this is driving all scoring right now with wrong data
2. **Create `article-digest.md`** — even a stub with 3-5 proof points from cv.md so modes have metrics to pull
3. **Create or replace `voice-dna.md`** — current file is Santiago's voice; decide if the user wants to adapt it or start fresh
4. **Add CV variants to `DATA_CONTRACT.md`** — declare them so they're update-protected
5. **Create `data/follow-ups.md`** — stub file so the followup mode doesn't error
